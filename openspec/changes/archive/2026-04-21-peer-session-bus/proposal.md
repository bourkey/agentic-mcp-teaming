## Why

The coordinator today spawns fresh `claude -p` subprocesses for every agent turn, which throws away context between turns and blocks the original intent of the project: letting multiple long-lived Claude Code sessions (tmux panes attached to git worktrees) team up through the coordinator. This change introduces a minimum viable peer-messaging bus so those live sessions can send typed events and short chat messages to each other. Scope has been deliberately narrowed after two review-gate rounds identified that a larger bus design accumulated too much surface (broadcasts, queue-to-unregistered, pluggable notifiers, retention, heartbeats, selective reads) — each of which brought its own failure modes. This proposal ships the smallest useful bus and defers everything else to follow-up changes.

## What Changes

- Add three MCP tools: `register_session`, `send_message`, `read_messages`. No `mark_read`, no `list_sessions`, no broadcast, no queueing to unregistered names in this change.
- `register_session({ name, priorSessionToken? })` returns an opaque `sessionToken` (256-bit random, base64url). Callers MUST present the token on every subsequent `send_message` and `read_messages` call; the coordinator validates via SHA-256 + `timingSafeEqual`. Re-registering an existing name requires the caller to present that name's currently-valid `priorSessionToken` as proof-of-continuity; only new names register without one. A client that loses its token (local restart) cannot recover ownership of an existing name from this API — operator intervention (removing the registry entry) is required. There is no `from` parameter on `send_message` or `read_messages`; identity is derived entirely from the token.
- Coordinator restart invalidates all previously-issued sessionTokens. Tokens live only in memory; every `tokenHash` loaded from `registry.json` at startup is cleared before the registry goes live. Clients re-register on coordinator restart using their own `name`; because no prior token is active, the coordinator accepts re-registration without `priorSessionToken` when the stored `tokenHash` is empty. Unread lists survive.
- Messages are direct-only: `to` MUST be a currently-registered session name. Sending to an unregistered name returns `recipient_not_registered`; there is no pending queue, no `"all"` broadcast.
- Message envelope persisted as one JSON line in `sessions/<coord-session>/messages.jsonl`. Timestamps are persisted as ISO-8601 in UTC (trailing `Z`), never with a non-zero offset; comparison and ordering use `Date.parse()` not raw-string compare. Receivers call `read_messages`, which atomically drains and returns all unread envelopes wrapped in `<peer-message from="…" kind="…" messageId="…">…</peer-message>`. There is no `since` parameter and no `mark_read` — read is total drain. Body content and attribute values are XML-escaped, and XML 1.0 illegal control characters (U+0000–U+0008, U+000B, U+000C, U+000E–U+001F) are stripped before escape.
- Body size capped at `PEER_BUS_MAX_BODY_BYTES = 65536` bytes per message (not configurable). Per-recipient unread-list length capped at `PEER_BUS_MAX_UNREAD = 10000` entries; sends that would exceed the cap return `mailbox_full`. `read_messages` response capped at `PEER_BUS_MAX_RESPONSE_BYTES = 1048576` bytes cumulative envelope; when the unread list exceeds this, the tool drains only what fits, keeps the rest unread, and returns `hasMore: true`.
- `kind` enum: `workflow-event | chat | request | response`. When `kind === "workflow-event"`, `body` MUST be a JSON object containing a required non-empty `event: string` field; all other fields are free-form. Other kinds take arbitrary body content within the size cap.
- Notifier: when `peerBus.notifier.tmuxEnabled` is true, every delivered message triggers a tmux banner on the target window (`execFile("tmux", ["display-message", "-t", <name>, <format>])`) and flips the window-status-style. The notifier is invoked AFTER the coordinator releases all per-session mutexes, so it does not block subsequent sends to the same recipient. Notifier substitutions strip `#`, backticks, `$`, `;`, `&`, `|`, and newlines from peer-controlled values to neutralise `tmux` format-language expansion (`#(…)`). No pluggable notifier interface in this change — tmux is the only implementation.
- Single-instance enforcement: at startup with `peerBus.enabled: true`, the coordinator acquires an exclusive lock on `sessions/<coord-session>/coordinator.lock` via `fs.openSync(path, 'wx')`, writing `pid=<n>`. The PID is diagnostic-only; the coordinator SHALL NOT steal the lock based on liveness checks. Cleanup is registered on `SIGINT`, `SIGTERM`, `exit`, `uncaughtException`, and `unhandledRejection`, and verifies `pid=<n>` matches `process.pid` before unlinking. POSIX `O_EXCL` semantics are assumed; operators on NFS-pre-v3, overlay, and other filesystems that do not provide atomic `O_EXCL|O_CREAT` SHALL NOT enable `peerBus`.
- Per-session async mutex serialises all bus-tool mutations keyed by session name so concurrent `read_messages` from the same session do not race. Implementers MUST acquire mutexes in lexicographic name order (caller-first on tie).
- Registry state persisted to `sessions/<coord-session>/registry.json` via temp-file-rename. Persisted fields: `name`, `tokenHash`, `registeredAt`, `lastSeenAt`, `unreadMessageIds`. On coordinator restart every loaded `tokenHash` is cleared. No retention, no TTL, no stale-session logic. The in-memory registry container is `Map<string, Entry>` (never a plain object) to preclude prototype pollution via hand-edited `registry.json`.
- Authentication iterates every registry entry on every call, computes SHA-256 of the presented token, and compares against each entry's `tokenHash` via `crypto.timingSafeEqual`. Entries with an empty `tokenHash` (post-restart, pre-re-register) compare against a fixed 32-byte zero sentinel so iteration cost does not leak session state. The coordinator SHALL accumulate a match boolean and only branch AFTER the full loop; early return is forbidden.
- New Zod `peerBus` block in `src/config.ts` with: `enabled: boolean` (default false); `notifier.tmuxEnabled: boolean` (default false); `notifier.displayMessageFormat: string` (default `"peer-bus: from {from} kind {kind}"`); `notifier.unreadTabStyle: string` (default `"bg=yellow"`). `.strict()` applied throughout; unknown keys rejected at load; format-string validated against a conservative character allowlist.
- Error responses follow the MCP SDK's `isError: true` convention with a stable error-code enum (`invalid_session_name`, `invalid_session_token`, `invalid_prior_session_token_required`, `invalid_recipient_name`, `recipient_not_registered`, `payload_too_large`, `mailbox_full`, `invalid_workflow_event_body`, `response_internal_error`).
- `sessionToken` SHALL NEVER appear in any log stream (`audit.log`, stdout, stderr, MCP SDK diagnostic output). Audit entries redact tokens and hash bodies.
- Existing `invoke_agent`, `invoke_reviewer`, consensus workflow, and filesystem tools are unchanged. The bus coexists; it does not route agent invocations.

## Capabilities

### New Capabilities
- `peer-session-bus`: transport-bound session identity, direct-DM messaging, envelope semantics, persistence, tmux notifier, single-instance lock, body-size and mailbox-length enforcement for long-lived Claude Code sessions communicating through the coordinator.

### Modified Capabilities
- `mcp-integration`: exposes the three new MCP tools; adds the `peerBus` Zod block with strict validation; standardises error-response shape and an error-code enum across the peer-bus tools.
- `agent-registry`: introduces a parallel session registry (runtime-populated, token-protected, `Map`-backed) that is structurally distinct from the configuration-declared agent registry.

## Impact

- **New code**: `src/server/tools/peer-bus.ts` (tool handlers), `src/core/session-registry.ts` (name → entry Map, per-session mutex, timing-safe authentication), `src/core/message-store.ts` (append-only JSONL reader/writer with UTC-Z timestamp helpers), `src/core/notifier-tmux.ts` (single-purpose fire-and-forget notifier), `src/core/coordinator-lock.ts` (startup lock with PID verification), `src/core/logger.ts` (thin `Logger` interface).
- **Modified code**: `src/config.ts` (Zod `peerBus` block), `src/server/index.ts` (conditional tool registration, lock acquisition at startup, exit-path handlers).
- **Config**: optional `peerBus` object in `mcp-config.json`; absent means disabled; `toolAllowlist` gains three new tool names.
- **Persistence**: new files `sessions/<coord-session>/messages.jsonl`, `registry.json`, `coordinator.lock`. Existing `state.json`, `audit.log`, `snapshots/` unchanged.
- **Security-sensitive surface**: one new subprocess call (`tmux`), invoked only via `execFile` with an argument array, with peer-controlled substrings passed through a format-sanitiser that strips tmux format sequences. Token comparisons use `timingSafeEqual` over every entry with no short-circuit. Registry loads into a `Map`, never a plain object.
- **Tests**: unit tests for store, registry (including mutex, timing-oracle, prototype-pollution resistance), notifier (mocking `execFile`), and tool handlers. Integration tests covering register → send → read flow end-to-end, token-required and priorSessionToken-required error paths, mailbox-full, response-size cap with `hasMore`, cross-addressee reconciliation on restart, and second-coordinator refusal.
- **Docs**: README gains a "Peer session bus" section; `CLAUDE.md` gains `execFile`-only, token-never-logged, and compile-time-cap reminders for this subsystem.
- **Explicitly deferred to later changes** (each gets its own proposal when the need is proven): `list_sessions`, `whoami`, `mark_read` / non-draining reads, `since` selective reads, broadcast (`to: "all"`), queueing to unregistered names, pending-queue retention/TTL, `deregister_session`, stale-session detection, pluggable notifiers (desktop/VS Code), message retention and log compaction, rate limiting, per-`kind` body-size tuning, long-poll `wait_for_message`, filesystem-watcher-driven workflow event emission, VS Code extension bus observer. The companion `sysdig-pov-docs` proposal will cover the launcher script, the session-startup skill, and the opsx-skill hooks that emit `workflow-event` messages.
