## Why

The peer-session-bus assumes its recipients are actively working when messages arrive. Today the `notifier-tmux` only decorates the target window's bar style (`set-window-option bg=yellow`) — a visual hint requiring an operator to attend the pane and prompt the session to run `read_messages`. This works for chained handoffs where the consumer is already mid-turn, but fails outright in the shape most peer-bus deployments actually exhibit: recipients idle between tasks, with no active turn to intercept inbound work. Live evidence during `coordinator-streamable-http` exploration surfaced repeated `notifier-tmux: set-window-option failed` warns for recipients `main`, `misc`, and `backend` — producers sending into a bus with no consumer loop to pick the work up. This change closes the producer-consumer liveness gap by letting the coordinator actively wake an idle recipient's tmux pane via a safe, operator-allowlisted `send-keys` payload, with a pane-state safety gate to prevent injection into interactive prompts.

## What Changes

- **New `wakeDispatcher` module**: auto-wake is extracted from the tmux notifier into a dedicated `peerBus/wakeDispatcher.ts` module that owns allowlist resolution, debounce, audit emission, pane-state safety gating, and failure handling. The dispatcher exposes a small `WakeBackend` interface; today's only implementation is `TmuxWakeBackend` living alongside `notifierTmux.ts`, leaving room for zellij/kitty backends without refactoring the dispatcher. The existing `notifier-tmux` passive window-bar decoration remains unchanged and fires on every message regardless of auto-wake opt-in.
- **Two `execFile` calls per wake** (on opt-in recipients): the `TmuxWakeBackend` SHALL make exactly two calls — first `tmux send-keys -l -t <target> <resolvedCommand>` (literal mode prevents key-name reinterpretation), then `tmux send-keys -t <target> Enter` (separate keystroke submits the prompt). Arguments passed as arrays to `execFile`; no shell interpolation anywhere.
- **Pane-state safety gate**: before dispatching `send-keys`, the `TmuxWakeBackend` SHALL invoke `tmux display-message -p -t <target> '#{pane_current_command}'` via `execFile` and compare the result against `peerBus.autoWake.allowedPaneCommands` (default `["claude", "bash", "zsh", "sh"]`). If the pane's current command is not in the allowlist, the wake SHALL be suppressed with a `wake_suppressed` audit entry (reason: `pane_state_unsafe`) and no keystrokes SHALL be sent. This prevents injection into interactive prompts (`sudo`, `git commit` in `$EDITOR`, pagers, permission dialogs) that would otherwise auto-confirm when the queued `Enter` lands.
- **`register_session` payload extension (flat scalar)**: accepts an optional `autoWakeKey: string` field. The value is a KEY into the operator-controlled `peerBus.autoWake.allowedCommands` map — never a free-form shell string. Rejection paths:
  - **Format-invalid key** (does not match `/^[a-zA-Z0-9_-]{1,64}$/`): rejected at the Zod boundary with error code `invalid_auto_wake_key` and a generic message that does NOT echo the rejected input (prevents log-pollution and info-leak channels).
  - **Well-formed but unknown key**: rejected at the Zod boundary with error code `invalid_auto_wake_key`; error message lists the valid allowlist keys but does not echo the rejected input.
  - **`autoWakeKey` supplied when `peerBus.autoWake` block is absent**: rejected by the handler (before schema evaluation, because no dynamic enum exists) with error code `auto_wake_disabled` and a clear message stating auto-wake is disabled on this coordinator.
  - **`autoWakeKey` supplied when `allowedCommands` is present but empty**: rejected by the handler with error code `auto_wake_disabled` (config is technically valid but nothing is allowable).
- **Config schema additions** (`src/config.ts`): new `peerBus.autoWake` block (promoted OUT of `peerBus.notifier` — auto-wake is backend-agnostic, notifier is tmux-specific; the nesting was wrong). Fields:
  - `allowedCommands: Record<string, string>` — map from short allowlist keys to the command strings to be injected. `.refine()` validates EACH value: non-empty after trim, no XML 1.0 illegal control chars (`\x00`–`\x08`, `\x0B`, `\x0C`, `\x0E`–`\x1F`, `\x7F`), no newlines (`\n`, `\r`), no bytes above `\x7E`, max length 512. `send-keys -l` prevents tmux key reinterpretation but NOT terminal control sequences — this config-schema scrub is the actual mitigation.
  - `defaultCommand?: string` — optional key into `allowedCommands`. Consumed ONLY when a registration supplies `autoWakeKey: null` (explicit opt-in with operator default); registrations that omit `autoWakeKey` entirely do NOT inherit the default.
  - `debounceMs: number` (default `1000`) — per-recipient debounce window.
  - `allowedPaneCommands: string[]` (default `["claude", "bash", "zsh", "sh"]`) — the pane-current-command allowlist for the safety gate above.
- **Registry schema**: `SessionRegistryEntry` gains optional `autoWakeKey?: string` (persisted to `registry.json`, revalidated against the current allowlist on load — entries referencing removed keys are cleared with a startup warn) and `lastWakeDispatchedAt?: number` (in-memory only, never persisted — restart-resetting the debounce window is intentional). The debounce state living ON the registry entry means concurrent `send_message` calls for the same recipient serialise through the existing per-session mutex — check-and-set of `lastWakeDispatchedAt` is atomic by construction, closing the same-tick race that a notifier-local map would leave open.
- **Audit event family `wake_*`**: three closed-family audit entries emitted by the wakeDispatcher:
  - `wake_dispatched`: a `send-keys` was actually emitted. Fields: `{ target, commandKey, messageId, dispatchedAt, status: "ok" | "failed", exitCode?, signal? }`.
  - `wake_suppressed`: dispatch was skipped. Fields: `{ target, commandKey, messageId, reason, suppressedAt }`. `reason` enum: `debounce | pane_state_unsafe | key_no_longer_in_allowlist`.
  - `wake_failed`: alias of `wake_dispatched` with `status: "failed"` — kept in the `wake_*` grep family by convention; implementations MAY emit a single `wake_dispatched` with `status: "failed"` OR a separate `wake_failed` entry (both satisfy the spec).
  None of these entries SHALL contain the resolved command string, the send-keys argv, any subset of `mcp-config.json`, session tokens, or the session `tokenHash`. The `commandKey` is the only allowlist-linked identifier that ever reaches the audit log.
- **Health counters**: the wakeDispatcher maintains three in-memory counters per recipient — `wakesDispatched`, `wakesSuppressed`, `wakesFailed` — readable through the existing audit-log aggregation layer (the same read path that today surfaces `lastSeenAt` for sessions). Zero on-wire cost. Gives operators a tuning signal for `debounceMs` and a health signal for "is auto-wake reaching pane targets at all?"
- **Sender-influence prohibition**: normative requirement that the `send-keys` argv SHALL be derived solely from (a) the registrant's stored `autoWakeKey` resolved through the operator-declared allowlist and (b) the target pane identifier. No field of the triggering `send_message` call (body, attributes, metadata, `messageId`, sender identity, headers) SHALL appear in the `tmux send-keys` arguments.
- **Capability boundary**: the `register_session` tool-contract additions (the new optional `autoWakeKey` field, the new error codes) and the config schema additions live in `specs/mcp-integration/spec.md` as `MODIFIED` requirements, matching where the archived bus proposal located tool contracts and the `peerBus` Zod block. The notifier/wakeDispatcher behaviour (safety gate, debounce semantics, audit-event family, failure handling, tmux target resolution) lives in `specs/peer-session-bus/spec.md`.
- **README recipe** ("Making your Claude Code session peer-reactive"): a new section documenting the complementary consumer-side pattern — a `Stop` hook running `coordinator read --wrap-for-prompt` to drain the inbox at turn boundaries, plus a `register_session`/`coordinator register --auto-wake <key>` invocation for idle wake-ups. The recipe is CLI-agnostic — it references the `allowedCommands` keys rather than a Claude-Code-specific slash command — and calls out explicitly that operators SHOULD only enable auto-wake on panes dedicated to the agent runtime (not on panes that also run `sudo`, `git commit`, or other interactive prompts). The `read_messages` tool is documented as idempotent-on-drain, so the Stop hook and the wake-injected inbox drain compose cleanly — whichever fires first drains the unread list, the other sees an empty inbox and is a no-op.

## Capabilities

### New Capabilities
_(none)_

### Modified Capabilities
- `peer-session-bus`: adds wakeDispatcher behaviour requirements (keyed-allowlist resolution, per-recipient debounce on the registry entry, pane-state safety gate, `wake_*` audit event family, failure handling, target resolution, sender-influence prohibition, health counters).
- `mcp-integration`: extends the `register_session` tool contract (new optional `autoWakeKey` input field, two new stable error codes, validation semantics) and extends the `mcp-config.json` schema with the new `peerBus.autoWake` block and its value-level refinements.

## Impact

- **Affected code**: new `src/peerBus/wakeDispatcher.ts` owning the active-injection state machine; new `src/peerBus/backends/tmuxWakeBackend.ts` implementing `WakeBackend` via `execFile` against tmux; `src/peerBus/notifierTmux.ts` reduced back to passive decoration; `src/session/registry.ts` gains `autoWakeKey` (persisted) and `lastWakeDispatchedAt` (in-memory); `src/config.ts` gains the `peerBus.autoWake` Zod block with value-level `.refine()`; `src/server/tools/peerBus.ts` (the `register_session` handler) validates the `autoWakeKey` against the loaded allowlist and surfaces the two new error codes.
- **Reused primitives**: `execFile`-only subprocess discipline for tmux invocations; the existing audit logger; the existing per-session registry mutex (now carrying debounce state, eliminating a race that a notifier-local map would have left open); the existing `loadConfig` + Zod error path for surfacing config violations.
- **New dependencies**: none. `tmux send-keys -l` and `tmux display-message -p '#{pane_current_command}'` are already part of the tmux CLI the notifier shells out to today.
- **Tests**: unit tests for the config schema value-level refinements (empty/control-char/newline/over-length rejection), for the `register_session` rejection paths (format-invalid, well-formed-unknown, empty-allowlist, block-absent), for the debounce (burst suppression, concurrent-different-recipients, same-tick atomicity, failure-consumes-window), for the pane-state safety gate (allowed → dispatch; not-allowed → suppress with `wake_suppressed` audit), for registry persistence (key survives restart, stale key cleared on load with warn), for audit entries (command key present, resolved string absent, failure status populated), for `send_message` success invariant regardless of wake outcome, for sender-influence prohibition (argv equality when send_message body contains shell metacharacters), and for health counters.
- **Documentation**: README recipe for the consumer-side Stop hook + auto-wake registration, including the safety warning about pane-dedicated use. `CLAUDE.md` update in the peer-bus security section: the `allowedCommands` allowlist is the authoritative boundary for injected content; `send-keys -l` prevents key reinterpretation but NOT terminal control sequences — the config-schema scrub is the real mitigation.
- **Inbound dependency**: the README recipe references `coordinator register --auto-wake <key>` from the in-flight `coordinator-client-cli` change. If `coordinator-client-cli` has not landed at merge time, the recipe SHALL fall back to documenting the raw MCP-tool `register_session` call with `autoWakeKey` until the CLI flag ships.

## Out of Scope (explicitly deferred)

- Per-message `urgent: true` override bypassing debounce (deferred until a real need appears).
- Server-initiated MCP push notifications via Streamable HTTP `GET /mcp` (Claude Code does not act on MCP notifications today).
- Hot-reload of `mcp-config.json` — operators changing the allowlist restart the coordinator, matching the behaviour for every other config field.
- Implementing the consumer-side `Stop` hook itself — the hook lives in each consumer's `.claude/settings.json` and is a documentation deliverable, not a code deliverable in this repo.
- Cross-multiplexer support (screen, zellij, kitty-tabs) — the `WakeBackend` interface accommodates future backends but this change ships only `TmuxWakeBackend`.
- A dedicated introspection HTTP endpoint (`GET /health`) for the counters — the audit-log aggregation layer is sufficient for v1.
