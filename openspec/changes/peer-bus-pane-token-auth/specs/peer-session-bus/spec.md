## MODIFIED Requirements

### Requirement: `register_session` authenticates via a stable `paneToken` credential

`register_session` SHALL accept a required `paneToken` parameter — a caller-supplied credential generated by the launcher and injected into the pane's environment as `COORDINATOR_SESSION_TOKEN`. The `paneToken` SHALL be validated before registry access: it SHALL be rejected with `invalid_pane_token_missing` if absent, empty, or longer than 512 bytes. The coordinator SHALL store `sha256(paneToken)` as `paneTokenHash` on the registry entry — a dedicated field separate from the rotating `tokenHash`. The `paneToken` hash comparison SHALL use `crypto.timingSafeEqual` — never a string equality check — and SHALL be performed even when the result will be ignored (e.g., TTL-eviction path) to prevent timing oracle attacks. `paneToken` SHALL be redacted to `"<redacted>"` in all audit log entries and SHALL NOT appear in warn/error log messages, Zod validation errors, or any MCP response field. `paneTokenHash` SHALL be persisted to `registry.json` (mode `0o600`) and SHALL survive coordinator restarts — it is never wiped by `clearTokenHashes()`. This enables seamless re-registration after a coordinator restart without operator intervention.

On `register_session({ name, paneToken })`, the coordinator SHALL apply the following logic:

1. **No existing entry**: store `sha256(paneToken)` and issue a new `sessionToken`. This is a fresh registration.
2. **Existing entry, hash matches**: issue a new `sessionToken`. The entry's `registeredAt` SHALL be preserved; `lastSeenAt` SHALL be updated.
3. **Existing entry, no `paneTokenHash`** (loaded from a pre-change `registry.json`): treat as unowned — store `sha256(paneToken)`, issue `sessionToken`, preserve `registeredAt`. This is the one-time migration path after coordinator upgrade.
4. **Existing entry, hash mismatch, `lastSeenAt` within `peerBus.session.inactivityTtlMs`**: reject with error code `invalid_pane_token`. The name is live and owned by a different pane.
5. **Existing entry, hash mismatch, `lastSeenAt` past TTL**: evict the stale entry, store new `sha256(paneToken)`, issue a new `sessionToken`. A warn SHALL be logged at eviction time naming only the evicted session name and timestamp — it SHALL NOT include `paneToken`, `paneTokenHash`, or any field derived from either.

The `priorSessionToken` parameter and `invalid_prior_session_token_required` error code are removed. If a caller supplies `priorSessionToken`, the coordinator SHALL silently ignore it (Zod strip behaviour).

#### Scenario: Fresh registration stores hash and issues token

- **WHEN** `register_session({ name: "claude-main", paneToken: "<secret>" })` is called and no entry exists for `claude-main`
- **THEN** the coordinator SHALL store `sha256("<secret>")` on the new entry and return a `sessionToken`; the `paneToken` value SHALL NOT appear in the registry, audit log, or any response field. Note: the raw `paneToken` value will be visible in LLM conversation context at registration time — this is accepted per the local dev tool threat model; the audit-log exclusion applies only to the coordinator-side audit log.

#### Scenario: Re-registration after compaction always succeeds

- **WHEN** `register_session({ name: "claude-main", paneToken: "<secret>" })` is called and an existing entry exists with a matching `sha256("<secret>")`
- **THEN** the coordinator SHALL issue a new `sessionToken`; `registeredAt` SHALL be unchanged; `lastSeenAt` SHALL be updated to now

#### Scenario: Legacy entry (no paneTokenHash) allows fresh registration

- **WHEN** `register_session({ name: "claude-main", paneToken: "<secret>" })` is called and an existing entry exists with no `paneTokenHash` field (loaded from a pre-change registry)
- **THEN** the coordinator SHALL store `sha256("<secret>")`, issue a new `sessionToken`, and preserve `registeredAt`; this entry is now fully migrated

#### Scenario: Mismatched token from a live session is rejected

- **WHEN** `register_session({ name: "claude-main", paneToken: "<wrong-secret>" })` is called and an existing entry exists with a non-matching hash AND `lastSeenAt` is within TTL
- **THEN** the coordinator SHALL return error code `invalid_pane_token` and SHALL NOT modify the existing entry or issue a token

#### Scenario: Mismatched token from a stale session triggers TTL eviction

- **WHEN** `register_session({ name: "claude-main", paneToken: "<new-secret>" })` is called and an existing entry exists with a non-matching hash AND `lastSeenAt` is past `peerBus.session.inactivityTtlMs`
- **THEN** the coordinator SHALL evict the old entry, store `sha256("<new-secret>")`, issue a new `sessionToken`, and log a warn containing only the evicted session name and timestamp — no `paneToken`, `paneTokenHash`, or derived values

#### Scenario: Missing or empty `paneToken` is rejected

- **WHEN** `register_session({ name: "claude-main" })` is called without a `paneToken`, or with an empty string, or with a value exceeding 512 bytes
- **THEN** the coordinator SHALL return error code `invalid_pane_token_missing` and SHALL NOT create or modify any registry entry

### Requirement: `peerBus.session.inactivityTtlMs` configures the stale-entry eviction window

`mcp-config.json` SHALL accept an optional `peerBus.session.inactivityTtlMs` field (non-negative integer, milliseconds). When absent, the coordinator SHALL apply a default of `600000` (10 minutes). An entry is considered stale when `(now - Date.parse(lastSeenAt)) >= inactivityTtlMs` — exactly at the boundary is stale.

When set to `0`, TTL eviction is disabled — stale entries are never evicted by inactivity alone. **Operators choosing `0` accept that pane replacement requires coordinator restart or manual registry edit to clear the existing `paneTokenHash`.**

#### Scenario: Default TTL applies when field is absent

- **WHEN** `mcp-config.json` does not include `peerBus.session.inactivityTtlMs`
- **THEN** the coordinator SHALL treat the TTL as 600000 ms; a mismatched-token registration against an entry last seen 11 minutes ago SHALL succeed via eviction

#### Scenario: TTL of 0 disables eviction for entries with a paneTokenHash

- **WHEN** `peerBus.session.inactivityTtlMs` is `0` AND an entry exists with a `paneTokenHash`
- **THEN** no entry SHALL ever be evicted by inactivity alone; a mismatched-token registration SHALL always return `invalid_pane_token` regardless of `lastSeenAt`

#### Scenario: TTL of 0 still allows registration against legacy entries (no paneTokenHash)

- **WHEN** `peerBus.session.inactivityTtlMs` is `0` AND an entry exists with no `paneTokenHash` (legacy pre-change entry)
- **THEN** `register_session` with any `paneToken` SHALL succeed as fresh registration semantics (case 3 above); TTL=0 only restricts hash-mismatch eviction, not unowned-entry takeover

#### Scenario: Custom TTL is respected

- **WHEN** `peerBus.session.inactivityTtlMs` is `60000` (1 minute) AND a mismatched-token registration arrives for an entry last seen 90 seconds ago
- **THEN** the coordinator SHALL evict the old entry and allow fresh registration

### Requirement: `peer-bus-reminder.sh` hook instructs LLM to read env token — never inlines raw value

The `peer-bus-reminder.sh` hook output SHALL instruct the LLM to read `$COORDINATOR_SESSION_TOKEN` from the environment when calling `register_session`. The raw token value SHALL NOT appear in hook output text. This ensures the token is absent from conversation history on every turn; it appears in LLM context only at registration time when the LLM reads it from the environment to construct the MCP call.

#### Scenario: Hook output never contains raw token

- **WHEN** the hook fires on any turn where `$COORDINATOR_SESSION_NAME` is set
- **THEN** the hook output SHALL contain an instruction referencing `$COORDINATOR_SESSION_TOKEN` by name (e.g. "read `$COORDINATOR_SESSION_TOKEN` from env") but SHALL NOT contain the actual token value

## REMOVED Requirements

### Requirement: `register_session` issues rotating session tokens requiring `priorSessionToken` for re-registration

**Reason**: The rotating `priorSessionToken` mechanism requires the previous token to be in LLM working context. Context compaction destroys it, making re-registration impossible without operator intervention. Replaced by the stable `paneToken` credential (see MODIFIED requirement above).

**Migration**: Replace `register_session({ name })` calls with `register_session({ name, paneToken: $COORDINATOR_SESSION_TOKEN })`. Remove all `priorSessionToken` handling from skills and workflow commands. Remove all `invalid_prior_session_token_required` error handling.
